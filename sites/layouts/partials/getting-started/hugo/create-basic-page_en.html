Generate New Page (Language)

Here's the first hestiaHUGO built-in feature: multi-lingual enabled by default without sacrificing root access. Unlike Hugo's built-in i18n feature, hestiaHUGO allows you to build pages and materials right at root-level despite using multilingual directory-oriented feature.


However, for this walkthrough, we won't be constructing these pages yet but to create them as placeholder. We will work on it in later chapters. To create a hestiaHUGO configured page, simply use the hugo new command with --kind argument: 

$ hugo new --kind hestia content/[URL_pathing_in_lowercase]

You are highly encouraged to setup your language directory from the get-go even when you're operating on a single language. This prevents the need of a massive URL redirection in the future when you attempt to support another language. hestiaHUGO does not dictate the language and country codes naming pattern so feel free to make your own. You will need them to configure in each page's language settings later on.


In this walkthrough, en and zh-hans were chosen so we ran the commands 3 times as follows: 

$ hugo new --kind hestia content/
$ hugo new --kind hestia content/en
$ hugo new --kind hestia content/zh-hans

Setup Page Settings

Once done, you can proceed to setup the page metadata (the __page.toml file). This file is responsible for a lot of the page critical metadata like title, description, keywords, redirect rules, timestamps, and list of data files to source from. To keep things basic and simple, we need to handle the following items:

  1. Date.Created - The date where the page is created. (e.g. on linux, $ date is the easiest way to generate a compatible one);

  2. Date.Published - The date where the page is published for public view;

  3. Content.Title - The title of the page. Keep it as short as 100 characters for SEO appearance;

  4. Content.Keywords - The keywords for the page. Keep the list concise and percise. Note that some SEO won't use this field anymore;

  5. Description.Pitch - A short pitch about the page. Keep it as short as 100 chracters for SEO appearance;

  6. Description.Summary - Continues and expand from the Description.Pitch. You can keep it as long as you want.

Setup Page SEO Robots Rules

The next thing to deal with is setting the SEO robots rules (the __robots.toml file). If you don't have special rules for the robots, you may skip this step.

However, since we are working on 404 page at the moment and we really do not want any robots to make it appear on any search listing, let's go ahead to add the robots' rules. For this demo, we just need to add 2: 

[[Meta]]
Name = 'googleBot'
Content = 'noindex, nofollow'

[[Meta]]
Name = 'robots'
Content = 'noindex, nofollow'

Create Page Content (HTML)

All right, it's time to construct the page content. Let's start with HTML format. On hestiaHUGO, we parse a different content file called __content.html complying to __content.[FORMAT] naming convention instead of the conventional Hugo's _index.html. That being said, editing the Hugo's _index.html file does nothing and the __content.[FORMAT] naming convention MUST HAVE 2 UNDERSCORE. Also, Markdown format is prohibited to keep things really simple and non-restrictive. There are a few reasons for doing this:

  1. We want to keep the learning curve and knowledge requirements back to basic: HTML+CSS+JS, Go's text/template formatter, and html/template formatter;

  2. Hugo's data path is very inefficient when processing its content page alongside with shortcodes (Page data is being re-created at least twice just for rendering 1 page output);

  3. Shortcode is a duplicate efforts of Hugo's partial template function and unnecessary horizontal duplication is very costly;

  4. A bunch of critical unresolved Hugo-specific problems that we rather not talk or tired of talking to the dev team so we take matters in our own hand and develop a standard library from scratch inside theme module;

  5. We're sick of Hugo's Shortcode Ã— Markdown frankensteining shenanigans where itself can easily mutate into its very own programming language and create unnecessary learning curve and maintenance works that are not portable.

Another advantage of deploying our own libraries is that one can use data-driven method to construct various outputs for multiple formats with high consistency with data filler clauses. To keep things simple for this demo (as in coaching a SEO capable HTML, CSS, and JS itself can be a full-time course), we keep the __content.html as little as the following but not restricting you to be fully creative: 

<main>
	<section id='introduction' class='banner'>
		<h1>{{- .Titles.Page -}}</h1>
		<p>{{- .Descriptions.Page.Pitch }} {{ .Descriptions.Page.Summary -}}</p>
	</section>
</main>

hestiaHUGO always provides a maximum control flexibility for you to build your own content page as you wish. It's basically a flexible and clean canva to work out your content. Anything you did here stays within the page. The other guides, we will show you how to share parts across different pages.


Hence, we start straight with the <main> elements. Then, as we build the introduction section where we used 3 data filler namely {{- .Titles.Page -}}, {{- .Descriptions.Page.Pitch }}, and {{ .Descriptions.Page.Summary -}} to automatically fills in the page settings' values we did earlier in the __page.toml file. Like Go's text/template package, the dash indicates whitespace removal on its side. You can obtain these data fields from the page's Hestia Debug Console where it displays all the keys and their associated values in it.

URL and Assets Placement

There is one more thing you have to understand about hestiaHUGO: it uses its own URL system complying to RFC3986 and most importantly, respecting your baseURL. Some publishers like GitLab Pages or GitHub Pages offer baseURL with pathing (e.g. project pages) and hestiaHUGO went further to make sure this works as originally intended. You need to remember 3 things:

  1. hestiaHUGO offers its own API absoluteURL to process your relative URL into the desired format;

  2. If your path begins with a slash (e.g. /en), it is relative to your baseURL; If your path begins without a slash (e.g. image.jpg), it is relative to your current page.

  3. For security reason, NEVER USE RELATIVE URLs for any published output URL; only use it for internal compilations (e.g. feed relative URL to absoluteURL to construct its absolute URL dynamically).

This also means that all your page-specific, non-sharing asset files are recommended to be placed inside your page directory. In this demo, let's place a graphic file into the /en/404 directory and deploy it as the page background. You can source your picture from Pexels.com. Then, upgrade your __content.html to be something as such: 

<main style='
	background: url({{- absoluteURL "html" "[FILENAME HERE]" -}})
		#000012
		no-repeat center center;
	background-size: cover;
'>
	<section class='banner' id='introduction'>
		<h1 style='text-align: center;'>{{- .Titles.Page -}}</h1>
		<p style='text-align: center;'>
			{{- .Descriptions.Page.Pitch }} {{ .Description.Page.Summary -}}
		</p>
		<br/>
		<a class='button' href='{{- .URL.Base -}}'>Head Back</a>
	</section>
</main>

Basically, we added the background image to the page (please remember to change the [FILENAME HERE] to your actual filename) and a "call to action" button to tell the lost souls for heading back to home page via inline CSS styling. Refresh the page once everything is done and you should have a simple and beautiful 404 page rendered nicely. At this point, you had completed your HTML output.
